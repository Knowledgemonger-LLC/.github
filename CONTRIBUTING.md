# Contributing

Thanks for your interest in contributing to a **Knowledgemonger-LLC** project. This is the
organization's default contributing guide; individual repositories may add their own on top of
it.

## Getting started

1. **Open an issue first** for anything non-trivial, so we can agree on the approach before you
   invest time. Small fixes (typos, obvious bugs) can go straight to a pull request.
2. **Fork and branch** from the default branch. Use a short, descriptive branch name.
3. **Keep changes focused.** One logical change per pull request makes review faster.

## Development

- Follow the setup and build instructions in the repository's own `README.md` / `AGENTS.md`.
- Match the existing code style. Most repos ship shared lint/format config and pre-commit hooks —
  run them before pushing.
- Add or update tests for behavior changes, and make sure the existing suite passes.

## Pull requests

- Fill out the pull request template.
- Write a clear description of **what** changed and **why**.
- Use [semantic commit](https://www.conventionalcommits.org/) messages where the repo uses them.
- Be responsive to review feedback; keep discussion on the PR so it stays discoverable.

## Reporting bugs and requesting features

Use the issue templates. For **security** issues, do **not** open a public issue — follow
[`SECURITY.md`](SECURITY.md) instead.

## Code of conduct

By participating, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md).
